# Data Layout

Top-level map of the `data/` tree. Every subdirectory has its own
`README.md` explaining what lives there and which files are load-bearing
for `bash ./scripts/validate_foundation.sh` ("foundation CI").

| Subdir | Role | Load-bearing? |
|---|---|---|
| `raw/` | Policy inputs and external source registries (no raw text). | Yes — `source_registry.json`, `source_scoring_rules.json` drive curation. |
| `external/` | Raw fetched text from authentic sources (Wikipedia, CC-100, Tatoeba, …). Gitignored if > 50 MB. | No — regenerable by `scripts/fetch_*.sh`. |
| `curated/` | Per-source Kazakh-only packs, plus training/evaluation manifests built from them. | **Yes** — `validate_foundation.sh` runs `jq empty` on every pack; manifests drive the training pipeline. |
| `tokenizer/` | BPE vocab, merges, and the frozen v0.3-v0.5 segmentation lexicon (211 curated roots + FST rules). | Yes. |
| `lexicon_v1/` | The authoritative v1 lexicon (~25.5 k roots: 13 606 pure Kazakh + 11 919 Apertium imports): Apertium import + Abai augmentation + purity audit. | Yes for `adam_fst` + `adam_chat`. |
| `training/` | Transformer training artifacts: latest checkpoint, perplexity reports, showcase generations, training manifests. Legacy from v0.4.0 transformer baseline — not exercised by v0.7.0+ dialog pipeline. | Mixed — see `training/README.md`. |
| `eval/` | Benchmark + tokenizer-experiment manifests, held-out datasets, delta reports, **`cognitive_dialog_dataset.json` (54 canonical scenarios + 0 aspirational at v4.4.5)**, **`repl_dialogs.json` (27 canonical + 3 aspirational dialogs at v4.4.6)**. The `benchmark_manifest.json` is a coverage / contract benchmark manifest (4 task families + guards + layers), **not** a single AI-benchmark score. | Yes. |
| `foundation/` | Top-level foundation-overview report aggregating all of the above. | Yes — regenerated by the CI script. |
| `dialog/` | v0.7.0+ dialog-layer templates (TOML) — **49 families** incl. v1.8.0 session-aware, v1.9.5 adapted-evidence, v2.7 `unknown.with_derived_chain`, v4.0.34 Tentative / Conflicted families, v4.3.4 four `ask_about_system.*` aspect families, v4.4.0 `dismiss_contradiction`, v4.4.5 `check_contradiction`. | Yes for `adam_chat`. |
| `world_core/` | v3.9.0+ human-authored Kazakh knowledge packs — **3424 entries / 4044 facts across 65 domains at v6.0.0-rc4** (rc4 base adds `kz_constitution.jsonl` 32 entries + `programming_java.jsonl` 30 entries + 25 individual `Қара сөздер` summaries; rc4-evening adds 6 chemical formulas + 2 Caspian/Mangystau geography + 1 Jupiter atmosphere fact to close `factual_eval_100` gaps; rc4 also fixes 73 entries that used unknown predicates and wires `validate_world_core` into `validate_foundation.sh`). All `approved` by `shaman`. **`geography_kz.jsonl` is the canonical source for the v4.3.0 `language_core::canonical_geo_entity` resolver.** | Yes — merged into `facts.json` by `extract_facts`; validated by `validate_world_core` (now a CI gate). |
| `retrieval/` | v1.6.0+ retrieval + reasoning artefacts: `morpheme_index.json`, `facts.json` (**4044 facts including text-extracted**), `lexical_graph.json`, `derived_facts.json` (**37 713 derivations** from 10 active rules). | Yes — consumed by `adam_chat`, `adam_demo`, `adam_inspect`. |
| `checkpoints/` | **v6.0.0-rc1.** L5.5 neural-composer checkpoints produced by `cargo run -p adam-agg-model --bin poc_kazakh_train`. Gitignored (regeneratable artefacts; 3-30 MB per checkpoint depending on hparams). Each checkpoint is a directory of `config.json` + `labels.json` + `training.json` + `model.mpk`. | Required for `adam_chat --neural-model <dir>` preview; optional for the deterministic kernel. |
| `scaling/` | v3.2.0+ scaling-bench reports: `scaling_report.json` across 5 tiers. | No — regenerable by `adam-scaling::scaling_bench`. |

## Load-bearing vs. generatable

- **Load-bearing** files are read from disk by binaries or CI scripts.
  Removing them breaks `validate_foundation.sh`.
- **Generatable** files are produced by scripts — safe to delete and
  rerun `scripts/run_*.sh` to regenerate.

When adding new data, prefer a dedicated subdirectory with a README
rather than dumping loose files at the root of `data/`.
