# Data Layout

Top-level map of the `data/` tree. Every subdirectory has its own
`README.md` explaining what lives there and which files are load-bearing
for `bash ./scripts/validate_foundation.sh` ("foundation CI").

| Subdir | Role | Load-bearing? |
|---|---|---|
| `raw/` | Policy inputs and external source registries (no raw text). | Yes — `source_registry.json`, `source_scoring_rules.json` drive curation. |
| `external/` | Raw fetched text from authentic sources (Wikipedia, CC-100, Tatoeba, …). Gitignored if > 50 MB. | No — regenerable by `scripts/fetch_*.sh`. |
| `curated/` | Per-source Kazakh-only packs, plus training/evaluation manifests built from them. | **Yes** — `validate_foundation.sh` runs `jq empty` on every pack; manifests drive the training pipeline. |
| `tokenizer/` | BPE vocab, merges, and the frozen v0.3-v0.5 segmentation lexicon (211 curated roots + FST rules). | Yes. |
| `lexicon_v1/` | The authoritative v1 lexicon (14 k entries): Apertium import + Abai augmentation + purity audit. | Yes for `adam_fst` + `adam_chat`. |
| `training/` | Transformer training artifacts: latest checkpoint, perplexity reports, showcase generations, training manifests. Legacy from v0.4.0 transformer baseline — not exercised by v0.7.0+ dialog pipeline. | Mixed — see `training/README.md`. |
| `eval/` | Benchmark + tokenizer-experiment manifests, held-out datasets, delta reports. | Yes. |
| `foundation/` | Top-level foundation-overview report aggregating all of the above. | Yes — regenerated by the CI script. |
| `dialog/` | v0.7.0+ dialog-layer templates (TOML). | Yes for `adam_chat`. |

## Load-bearing vs. generatable

- **Load-bearing** files are read from disk by binaries or CI scripts.
  Removing them breaks `validate_foundation.sh`.
- **Generatable** files are produced by scripts — safe to delete and
  rerun `scripts/run_*.sh` to regenerate.

When adding new data, prefer a dedicated subdirectory with a README
rather than dumping loose files at the root of `data/`.
