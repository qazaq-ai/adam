# Curated Data

Source-specific Kazakh packs (cleaned single-column text) and the
manifests that assemble them into training/validation sets.

## Source packs (read by `scripts/validate_foundation.sh`)

### Unfiltered (raw per-source imports)

Schema-checked by `jq empty` in CI, consumed by `lexicon_purity_audit`
and `extract_root_candidates` binaries.

|---|---|---:|---|

| `cc100_kk_pack.json` | CC-100 Kazakh web-crawl | 50,000 | CC-BY-SA 4.0 |
| `common_voice_kk_pack.json` | Mozilla Common Voice KK | 6,108 | CC0-1.0 |
| `kazakh_proverbs_pack.json` | Hand-curated proverbs | 280 | Public domain |

### Filtered (drop loanwords + 2-3-word fragments)

Produced by `cargo run --bin filter_pack` (v0.5.0 era). These are the
**actual training inputs** — `adam_training_corpus_manifest.json`
lists them, so `scripts/run_unified_corpus_assembly.sh` regenerates
`adam_training_corpus_pack.json` from filtered versions, not raw.

**Do not delete either set without also updating every manifest that
references them.** Both unfiltered (`jq empty` gates) and filtered
(`adam_training_corpus_manifest.json` assembly) are load-bearing in
`validate_foundation.sh`.

## Manifest hierarchy

The manifests chain together. Each one is a JSON document listing which
packs (or upstream manifests) feed it.

```
corpus_manifest.json                  foundation-level: every source pack
        │
        ├─ clean_training_corpus_manifest.json
        │       │
        │       └─ clean_training_corpus_pack.json        (assembled, deduped)
        │                   │
        │                   ├─ adam_training_corpus_manifest.json
        │                   │       │
        │                   │       ├─ adam_training_corpus_pack.json
        │                   │       ├─ adam_pretokenized_corpus_pack.json
        │                   │       ├─ adam_training_ids_pack.json
        │                   │       └─ adam_validation_ids_pack.json
        │                   │
        │                   ├─ mini_clean_training_manifest.json   (ablation)
        │                   │       └─ mini_clean_training_pack.json
        │                   │
        │                   └─ tiny_clean_training_manifest.json   (ablation)
        │                           ├─ tiny_clean_training_pack.json
        │                           ├─ tiny_clean_education_pack.json
        │                           ├─ tiny_clean_general_pack.json
        │                           └─ tiny_clean_reference_pack.json
```

### Which manifests to edit

- **`corpus_manifest.json`** — touch when adding a NEW source (e.g.
  Kazakh literature corpus). Ripples downstream.
- **`adam_training_corpus_manifest.json`** — touch when changing the
  training/validation split or the main training pack composition.
- **`mini_clean_training_manifest.json`**, **`tiny_clean_*`** — these
  are fixed experiment-matrix snapshots from v0.3–v0.4. Do not edit
  unless regenerating the entire ablation suite.

## Reports (generated by foundation scripts)

- `source_acceptance_*_report.json` — per-source quality audit
- `lexicon_expansion_report.json` — from Apertium + Abai augmentation
- `root_candidates_report.json` — OOV root extraction

These are regenerable and can be rebuilt via `scripts/run_*.sh`.

## Legacy context

The project pivoted from transformer training (v0.3–v0.4) to the
deterministic FST + dialog pipeline (v0.7.0+). The `adam_*_pack.json`
files and `clean_*_pack.json` ablation splits are legacy from that
pivot. They are kept because `validate_foundation.sh` still exercises
them as regression surface — do not delete.
