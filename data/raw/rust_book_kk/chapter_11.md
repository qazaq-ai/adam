# 11-тарау. Автоматты сынақтар жазу

Сынақ — кодты жалғастырмастан бұрын оның дұрыстығын тексеретін
бағдарлама. Әр жаңа мүмкіндік қосылғанда, бар сынақтар сізге кодыңыздың
ескі мінезі жоғалмағанын мойнына алады. Олардың нәтижесінде сіз дауысты
түрде «менің кодым жұмыс істеп тұр» дей аласыз — қарап тұрып емес,
сынақтармен дәлелдеп.

`Rust`-та сынақ жазу тілдің құрамдас бөлігі. Жаңа жобада дайын
дайындаулар бар: `cargo test` командасымен әр жаңартудан кейін сынақтар
жүгіртіледі; жаңа функцияға арналған сынақты жазу — бір атрибут пен
бір функция.

Бұл тарауда үш бөлік бар:

1. **Сынақты қалай жазу** — `#[test]` атрибуты, `assert!`-дер тобы,
   panic-күтілуі, тестілерде `Result` қолдану.
2. **Сынақтардың жұмысын қалай басқаруға болады** — параллельді / дәйекті,
   шығаруларды көру, ішкі жиынды жүгірту, елемеу.
3. **Сынақтарды ұйымдастыру** — бірлік тесттері vs интеграциялық
   тесттер; жасырын функцияларды тестілеу.

## 11.1. Сынақты қалай жазу

`Rust`-та сынақ — әдеттегі функция, бірақ оның үстінде `#[test]`
атрибуты тұрады. `cargo test` командасы атрибутпен белгіленген барлық
функцияларды табады, оларды бөлек жүгіртеді және әрбірінің сәтті өтуін
немесе кетпеуін тіркейді.

### Сынақтың құрылымы

`cargo new adder --lib` командасымен жаңа кітапхана сандығын жасайық.
Cargo сізге дайын тестпен `src/lib.rs` файлын ұсынады:

```rust
pub fn add(left: usize, right: usize) -> usize {
    left + right
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn it_works() {
        let result = add(2, 2);
        assert_eq!(result, 4);
    }
}
```

Жолма-жол қарастырайық:

- `#[cfg(test)]` атрибуты — модулді тек `cargo test` ішінде құрастыруды
  білдіреді. Кәдімгі құрастыруда (`cargo build` / `cargo run`) бұл модуль
  орындалатын файлға қосылмайды. Сондықтан тест коды оқырманға қол
  жетімді, әрі шығарылым бағдарламаны үлкейтпейді.
- `mod tests { ... }` — тесттерге арналған модуль. Шарт емес, бірақ —
  идиоматикалық.
- `use super::*;` — ата модульдің барлық элементтерін аумаққа енгізеді.
  Біздің жағдайда — `add` функциясын.
- `#[test]` атрибуты — функцияны сынақ ретінде белгілейді.
- `assert_eq!(result, 4)` — макрос, екі мәнді теңдеу. Тең емес болса —
  panic-қа барады, сынақ кетеді.

`cargo test` орындасаңыз, шығатын:

```text
running 1 test
test tests::it_works ... ok

test result: ok. 1 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out
```

Cargo сізге қанша сынақ өткенін, қанша кеткенін, қанша еленбегенін
көрсетеді.

### Сынақтың сәтсіз өтуі

Сынақ panic-қа барса — ол кетеді. Әдейі бір кететін сынақ қосайық:

```rust
#[test]
fn another() {
    panic!("Бұл сынақ әдейі кетеді");
}
```

`cargo test` шығаратыны:

```text
test tests::another ... FAILED

failures:

---- tests::another stdout ----
thread 'tests::another' panicked at 'Бұл сынақ әдейі кетеді', src/lib.rs:11:9
```

Сынақтардың бір-бірінен айырмасы жоқ — біреу кетсе, басқалары жалғаса
береді. Жалпы қорытынды соңында: барлық сынақтар тізімі, кеткені және
себептер.

### `assert!` макросы

`assert!(condition)` — шарт `true` болуын тексереді. `false` болса —
panic.

```rust
#[test]
fn larger_can_hold_smaller() {
    let larger = Rectangle { width: 8, height: 7 };
    let smaller = Rectangle { width: 5, height: 1 };
    assert!(larger.can_hold(&smaller));
}
```

`assert!` бір аргумент алады — логикалық өрнек. Әсер мақсатына сай — кез
келген өрнекті тексеруге болады.

### `assert_eq!` пен `assert_ne!`

`assert_eq!(left, right)` — екі мәннің тең болуын тексереді.
`assert_ne!(left, right)` — тең *болмауын*. Қарапайым `assert!(a == b)`-тан
артықшылығы — кетсе, екі мәнді де экранға шығарады:

```rust
#[test]
fn it_adds_two() {
    assert_eq!(add(2, 2), 5);
}
```

Шығатыны:

```text
---- tests::it_adds_two stdout ----
thread 'tests::it_adds_two' panicked at 'assertion `left == right` failed
  left: 4
  right: 5'
```

`left` — нақты мән, `right` — күтілген. Кеткенде дереу қатенің көлемі
анық: 4 келді, 5 күтілді. `assert!`-пен бұл дәл осындай ыңғайлы емес
болар еді.

`assert_eq!` мен `assert_ne!` мәндерді салыстыру үшін `PartialEq`
трейтін, ал басып шығару үшін `Debug` трейтін талап етеді. Сондықтан
сіздің өзіңіздің құрылымдарыңызды салыстыруға дайындау үшін
`#[derive(PartialEq, Debug)]` қосу керек.

### Қате хабарламасын қосу

Әдепкі хабарлама нақты болмаса — өзіңіздікін қосуға болады:

```rust
#[test]
fn greeting_contains_name() {
    let result = greeting("Айнұр");
    assert!(
        result.contains("Айнұр"),
        "Хабарлама атпен басталмады. Нақты: `{}`",
        result
    );
}
```

`assert!`-тің екінші аргументі — `format!`-қа ұқсас формат жолы.
`assert_eq!` мен `assert_ne!`-та да дәл солай.

### `should_panic` арқылы panic-ты тексеру

Кейде дұрыс мінез — қатеге panic-қа бару. Соны тексеру керек болса
`#[should_panic]` атрибутын қосыңыз:

```rust
pub struct Guess {
    value: i32,
}

impl Guess {
    pub fn new(value: i32) -> Guess {
        if value < 1 || value > 100 {
            panic!("Болжам 1 мен 100 арасында болуы керек, {} келді.", value);
        }
        Guess { value }
    }
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    #[should_panic]
    fn greater_than_100() {
        Guess::new(200);
    }
}
```

Сынақ panic-қа бармаса — кетеді. Panic-қа барса — өтеді.

`should_panic` тым кең болуы мүмкін — кез келген panic сай келеді,
тіпті сіз күтпеген. Нақтырақ ету үшін `expected` параметрі бар:

```rust
#[test]
#[should_panic(expected = "1 мен 100 арасында")]
fn greater_than_100_strict() {
    Guess::new(200);
}
```

Сынақ panic хабарламасында «1 мен 100 арасында» деген сөз болғанда ғана
өтеді. Басқа себептен panic болса — кетеді.

### Тестілерде `Result<T, E>`

Сынақ функциясы `Result<T, E>` қайтаратындай етіп жазуға болады:

```rust
#[test]
fn it_works() -> Result<(), String> {
    if 2 + 2 == 4 {
        Ok(())
    } else {
        Err(String::from("екі қосу екі төрт болмады"))
    }
}
```

Бұл стиль `?` операторы арқылы көп әрекетті тізбектегенде ыңғайлы.
Бірақ `should_panic` атрибутын `Result` қайтаратын сынаққа қолдануға
болмайды — себебі сынақ `Err` қайтаруы керек, сондықтан panic-тан тыс
шарт өзінде бар.

## 11.2. Сынақтардың жұмысын басқару

`cargo test` нөлдік параметрлерсіз барлық сынақтарды бөлек ағындарда
параллельді жүгіртеді. Бұл әдеттен — қажетсіз болғанда — өзгертуге
болады.

`cargo test` командасы екі жалаушаны бөліп қабылдайды: біреуі — `cargo
test`-тің өзіне, екіншісі — сынақ бинарлы файлына (test runner-ге).
Ажырату — `--`. Мысалы:

```sh
$ cargo test            # әдеттегі шақыру
$ cargo test --help     # cargo test жалаушалары
$ cargo test -- --help  # test runner жалаушалары
```

### Параллельді vs дәйекті жүгірту

Әдепкі бойынша сынақтар — параллельді. Бұл — жылдам, бірақ:

- ортақ файлмен жұмыс істейтін сынақтар бір-бірін кедергі жасауы мүмкін;
- ортақ ортаға айнымалылар әсер етуі ықтимал;
- буфер шығармаларынан бөтен әсер шығып кетеді.

Параллельдікті өшіру:

```sh
$ cargo test -- --test-threads=1
```

Барлық сынақтар бірден бір ағында дәйекті жүгіреді. Баяу болады, бірақ
бір-біріне әсер ету мүмкіндігі жоқ.

### Шығаруларды көру

Әдепкі бойынша өткен сынақтың `println!`-ден шыққан мәтіні басылмайды
— тек сынақтың өту/кетуі көрсетіледі. Кеткен сынақтың шығарулары
басылады.

Барлық шығаруды (өткендердікі де) көру керек болса:

```sh
$ cargo test -- --show-output
```

### Сынақ атымен таңдау

Барлық сынақты емес, тек белгілі бір аттарды жүгірту керек болса —
атау `cargo test`-ке аргумент ретінде беріледі:

```sh
$ cargo test it_works         # тек it_works сынағы
$ cargo test add              # аты `add`-пен басталатын барлығы
$ cargo test tests::it        # модуль ішінде «it» сөзі
```

Бұл — substring-сай: атын құрамына ұқсас барлық сынақтар жүгіреді.

### Сынақтарды елемеу

Кейде белгілі бір сынақ ұзақ — әр шакыруда жүгіртудің керегі жоқ. Соған
`#[ignore]` атрибуты:

```rust
#[test]
#[ignore]
fn expensive_test() {
    // 30 секунд алады...
}
```

`cargo test` командасы оны өткізіп жібереді. Тек белгіленген сынақтарды
жүгірту керек болса:

```sh
$ cargo test -- --ignored
```

Барлық сынақтарды (елемегендерді қоса) жүгірту:

```sh
$ cargo test -- --include-ignored
```

Бұл — ыңғайлы тірі-тіріге: жылдам сынақтарды әр өзгертіс кейін, ұзақ
сынақтарды (немесе ауыр сценарийлерді) тек CI-да немесе арнайы команда
бойынша жүгіртесіз.

## 11.3. Сынақтарды ұйымдастыру

`Rust` қауымдастығы сынақтарды екі үлкен топқа бөледі: **бірлік
тесттері** (unit tests) және **интеграциялық тесттер** (integration
tests).

- **Бірлік тесттері** — бір модульге шектелген, ішкі логиканы
  тексереді. Жасырын функцияларды да тексере алады. Сол модуль ішінде
  `#[cfg(test)]` мен `mod tests` блогы арқылы орналастырылады.
- **Интеграциялық тесттер** — кітапханаңызды сыртқы пайдаланушы сияқты
  қолданады. Жобаның түбіріндегі `tests/` қалтасында өздеріндік файл
  ретінде болады.

Екеуінің мақсаты әртүрлі. Бірлік тесттері — ішкі модульдердің
ұсақ-түйекті жұмысын тексереді. Интеграциялық тесттер — кітапхананың
жалпы интерфейсі дұрыс жұмыс істейтінін, бөліктер бір-бірімен бірге
жұмыс істейтінін.

### Бірлік тесттері

Бірлік тесттері — нақты модулмен бір файлда тұрады:

```rust
pub fn add_two(a: i32) -> i32 {
    internal_adder(a, 2)
}

fn internal_adder(a: i32, b: i32) -> i32 {
    a + b
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn internal() {
        assert_eq!(internal_adder(2, 2), 4);
    }
}
```

`internal_adder` — жасырын функция (модульден сыртқа `pub` етілмеген).
Бірақ `tests` модулі сол модульдің ішінде тұрғандықтан — оған еш
қиналмай қолжетімді. Бұл — бірлік тестілердің мықты жағы: ішкі
логиканы сыртқа шығармай тексеруге болады.

`#[cfg(test)]` атрибуты — компилятордың сынақ модулін тек `cargo test`
ішінде құрастыруды кепілдейді.

### Интеграциялық тесттер

Интеграциялық тесттер — кітапхана сандығын сыртқа таныстырғандай
қолданатын жеке файлдар. Олар жобаның түбіріндегі `tests/` қалтасында
тұрады. Cargo сол қалтаны автоматты түрде интеграциялық тесттер деп
қарастырады.

Алдыңғы мысалымыздағы `adder` сандығына интеграциялық тест қосайық:

```text
adder/
├── Cargo.toml
├── src/
│   └── lib.rs
└── tests/
    └── integration_test.rs
```

`tests/integration_test.rs`:

```rust
use adder::add_two;

#[test]
fn it_adds_two() {
    assert_eq!(add_two(2), 4);
}
```

Әрбір `tests/` файлы — өзінің бөлек сандығы. Сондықтан кітапхананы
`use adder::add_two` арқылы жариялаймыз — `super` арқылы емес.
Жасырын функцияларға кірудің мүмкіндігі жоқ — тек ашық
интерфейс.

`cargo test` шығатыны:

```text
running 1 test
test internal ... ok

test result: ok. 1 passed; 0 failed
   ...
running 1 test
test it_adds_two ... ok

test result: ok. 1 passed; 0 failed
```

Cargo бірлік тесттері мен интеграциялық тесттерді бөлек шығарады.
Бір немесе екеуі кетсе — қайсысы екені анық.

### Интеграциялық тесттерде ішкі модуль

Бірнеше интеграциялық тест ортақ көмекші кодты қолданғыңыз келсе —
жай ғана `tests/common.rs` жасасаңыз — ол өзімен жеке тест ретінде
жүгіртіледі. Бұл — кейде ыңғайсыз. `Rust` шарты бойынша — ортақ кодты
`tests/common/mod.rs` файлында ұстайды:

```text
tests/
├── common/
│   └── mod.rs
└── integration_test.rs
```

`tests/common/mod.rs`:

```rust
pub fn setup() {
    // ортақ дайындау логикасы...
}
```

`tests/integration_test.rs`:

```rust
use adder::add_two;

mod common;

#[test]
fn it_adds_two() {
    common::setup();
    assert_eq!(add_two(2), 4);
}
```

`tests/common/mod.rs` стилі — Cargo-ға бұл файл өздігімен тест емес,
тек ортақ модуль екенін айтады.

### Екілік сандықтар үшін интеграциялық тесттер

Жобаңыз тек `src/main.rs`-те екілік сандық болса — `tests/`
қалтасынан функциялар импорттай алмайсыз. Сондықтан `Rust`
қауымдастығы — кодтың көп бөлігін `src/lib.rs`-ке шығарып,
`src/main.rs` тек оны шақыратын тонкый қабат жасайды:

```rust
// src/main.rs
fn main() {
    if let Err(e) = my_app::run() {
        eprintln!("Қате: {}", e);
        std::process::exit(1);
    }
}
```

Бар басты логика `src/lib.rs`-та. Сол кітапхана сандығына `tests/`
ішінен — әдеттегі жолмен — интеграциялық тест жазуға болады.

## 11.4. Тарау қорытындысы

`Rust`-тың сынақ инфрақұрылымы — қарапайым әрі күшті. `#[test]`
атрибутымен функцияны белгілейсіз; `assert!`-дер тобымен шарттарды
тексересіз; `should_panic` арқылы күтілетін қателерді тексересіз;
`Result<T, E>` сынақтарымен `?`-операторды қолдана аласыз.

`cargo test` командасы сынақтарды параллельді жүгіртеді — әдеттен
тыс жағдайларда оны өзгертуге болады. Сынақ атымен таңдау,
`#[ignore]` арқылы ұзақ сынақтарды кейінге қалдыру, `--show-output`

кепілдейді. Бірлік тесттері модульмен бірге; интеграциялық тесттер
`tests/` қалтасында жеке файлдарда.

Келесі тарауда осы үйренгенімізді практикада қолданып, шынайы
бағдарлама — мини-`grep` — жазамыз. Сол арқылы қалай тиісті жобаны
бастан бастап жасауға, қателерді өңдеуге, кодты модульдерге бөлуге,
сынақтармен қорғауға болатынын көреміз.
