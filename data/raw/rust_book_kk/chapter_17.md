# 17-тарау. `Rust`-тың объектілі-бағытталған мүмкіндіктері

Объектілі-бағытталған бағдарламалау (ОО) — кодты ұйымдастырудың
тарихи жиі қолданылатын тәсілі. Smalltalk, Java, C++, Python, Ruby —
бәрі осы парадигма дәстүрі. `Rust` — ОО тілі емес, бірақ оның
типтік жүйесі көп ОО-үлгіні **басқа жолдармен** жасауға мүмкіндік
береді.

Бұл тарауда:

1. ОО тілдерінің ерекшеліктері. `Rust` оларды қалай беретінін, не үшін
   беретінін, қайсысы оның дизайнына сай емес екенін.
2. **Трейт-нысандар** (trait objects) — әр түрлі типтегі мәндерді бір
   ұжымда ұстайтын құрал.
3. ОО **жобалау үлгісінің** мысалы: **күй үлгісі** (state pattern).
   Оны `Rust`-тың типтік жүйесімен қалай жұмсартуға болатынын.

`Rust`-тың тұрғысы: «бар проблеманы шеш, оның үшін типке арналған
нақты механизмді қолдан». ОО-парадигма үшін «инеленемесе» барлық —
тек кейбірі.

## 17.1. Объектілі-бағытталған тілдердің сипаттамалары

ОО тілдеріне ортақ қандай сипат? Ең әйгілі дәстүрлі дереккөз — Gang
of Four кітабы (Design Patterns, 1994) — мынаны айтады:

> Объектілі-бағытталған бағдарламалар нысандардан құрылады. Нысан
> деректер мен тәртіптерді қамтиды.

`Rust`-тың `struct` пен `enum` — деректер; оларға бекітілетін `impl`-те —
тәртіп. Сонымен `Rust`-та нысандарға ұқсас не бар. Өзін «толық ОО»
деп атау керек пе? Бұл — терминологиялық сұрақ. Маңыздысы — `Rust`-тың
қандай тұжырымдамаларды қалай беретіні.

### Тұсаулау

**Тұсаулау** (encapsulation) — нысанның ішкі мәліметтері сыртқа
жасырын болатыны. Сыртқы код нысанмен оның ашық интерфейсі арқылы
ғана жұмыс істей алады.

`Rust`-та тұсаулау `pub` кілтсөзі арқылы:

```rust
pub struct AveragedCollection {
    list: Vec<i32>,
    average: f64,
}

impl AveragedCollection {
    pub fn add(&mut self, value: i32) {
        self.list.push(value);
        self.update_average();
    }

    pub fn remove(&mut self) -> Option<i32> {
        let result = self.list.pop();
        if result.is_some() {
            self.update_average();
        }
        result
    }

    pub fn average(&self) -> f64 {
        self.average
    }

    fn update_average(&mut self) {
        let total: i32 = self.list.iter().sum();
        self.average = total as f64 / self.list.len() as f64;
    }
}
```

`AveragedCollection` өрістері — `list` пен `average` — жасырын.
`add`, `remove`, `average` — ашық интерфейс. Сыртқы код тек осы үш
әдіс арқылы кіреді. Ішкі `update_average` — жасырын; орташа мән
үнемі дұрыс ұсталынуын сақтайды.

`Rust` тұсаулауды толық қолдайды. ОО-парадигмаға ұқсас.

### Мұрагерлік: жоқ

**Мұрагерлік** (inheritance) — бір типтің басқа типтен мінез-құлықты
автоматты түрде «алуы». Java-да `class A extends B` — `A` `B`-нің
барлық әдістерін мұрагерлікпен алады.

`Rust`-та осындай мұрагерлік **жоқ**. Ол шешім — саналы. Себептер:

1. **Мұрагерлік жиі шамадан көп бөлектерді біріктіреді.** Сізге
   `B`-ден тек бір әдіс керек болуы мүмкін, бірақ мұрагерлік арқылы
   барлығы келеді — қажетсіздіктерімен бірге.
2. **Мұрагерлік иерархия қатаюы.** Уақыт өте келе тарих беру арқылы
   құрылған жүйенің табиғаты қозғалады, бірақ қатаң иерархияны
   өзгерту қиынырақ.

`Rust`-та оны екі балама арқылы шешеді:

- **Әдепкі іске асыру** — трейттердегі әдіске әдепкі дене жазуға
  болады. Бекітіп алған тип — оны қолданады, немесе өзіндік нұсқасын
  жазып, оны ауыстырады.
- **Трейт-нысан** — әр түрлі типтерді бір интерфейс арқылы біріктіру.

Бұлар бірге — мұрагерліктің барлық пайдалы жақтарын береді, тіке
мұрагерлік құрылымынсыз.

### Көп пішінділік

**Көп пішінділік** (polymorphism) — бір интерфейспен әр түрлі типтерді
ұстау мүмкіндігі. Жалпы тілмен айтсақ — «бір атаумен бірнеше тип».
Көп ОО тілдерде көп пішінділік тек мұрагерлікпен бірге келеді.
`Rust`-та екеуі — бөлек:

- **Жалпылама типтер** — статикалық полиморфизм (компиляцияда
  әр нақты тип үшін мономорфтау).
- **Трейт-нысандар** — динамикалық полиморфизм (орындалу кезінде
  таратпа).

Бір кодты қай типтермен жасайтыныңыз — соған қарай таңдау.

## 17.2. Трейт-нысандар арқылы әр түрлі типтердегі мәндерді ұстау

Айталық, GUI кітапханасы жасап жатсыз. Сізде тізім бар: ол кез келген
GUI элементін — батырмаларды, жолды-енгізу алаңдарын, белгілерді,
суреттерді — ұстауы керек. Барлық элементтер бір ортақ әдіске ие —
`draw()`. Бірақ түрлі типтер.

Бір ұжымда `Vec<i32>` сияқты әр элемент бір типте болуы керек. Ал
бізге қажет — әр түрлі тип, бірақ ортақ интерфейс. Шешімі —
**трейт-нысан**.

### Ортақ тәртіпке арналған трейт жариялау

```rust
pub trait Draw {
    fn draw(&self);
}
```

Бұл — `Draw` трейті. Барлық GUI элементтері соны бекітуі керек.

### Трейт-нысандардың ұжымы

```rust
pub struct Screen {
    pub components: Vec<Box<dyn Draw>>,
}

impl Screen {
    pub fn run(&self) {
        for component in self.components.iter() {
            component.draw();
        }
    }
}
```

`Box<dyn Draw>` — **трейт-нысан**. Ол: «`Draw` трейтін бекітетін
қандай да бір тип, оның дәл түрі компиляцияда белгісіз» дегенді
білдіреді. `dyn` кілтсөзі — динамикалық таратпа.

Тізім — `Vec<Box<dyn Draw>>` — әр түрлі типтерді біріктіре алады.
Әр элемент жадта `Draw` трейтінің виртуальды кесте (vtable)-сімен
сақталады.

### Трейтті бекіту

```rust
pub struct Button {
    pub width: u32,
    pub height: u32,
    pub label: String,
}

impl Draw for Button {
    fn draw(&self) {
        println!("Батырма: {}", self.label);
    }
}

pub struct SelectBox {
    pub width: u32,
    pub height: u32,
    pub options: Vec<String>,
}

impl Draw for SelectBox {
    fn draw(&self) {
        println!("Таңдау: {:?}", self.options);
    }
}
```

Енді `Button` мен `SelectBox` екеуі де `Draw`-ды бекіткен. Оларды бір
ұжымға қосуға болады:

```rust
fn main() {
    let screen = Screen {
        components: vec![
            Box::new(SelectBox {
                width: 75,
                height: 10,
                options: vec![
                    String::from("Иә"),
                    String::from("Жоқ"),
                    String::from("Бәлкім"),
                ],
            }),
            Box::new(Button {
                width: 50,
                height: 10,
                label: String::from("OK"),
            }),
        ],
    };

    screen.run();
}
```

`screen.run()` — әр элементте `draw()` шакырады. Орындалу кезінде
`Rust` нақты типтің vtable-індегі сай функцияны табады және орындайды.

### Динамикалық таратпа vs статикалық таратпа

Бір сұрақ туады: неліктен `Box<dyn Draw>` емес, `<T: Draw>` жалпылама
тип параметрін қолданбасқа?

```rust
pub struct Screen<T: Draw> {
    pub components: Vec<T>,
}
```

Бұл нұсқа жұмыс істейді — бірақ айырмашылық:

- **`Vec<T>` (жалпылама)** — `T` бір нақты тип. Ұжым бір ғана типтегі
  элементтерге шектелген (мысалы, тек `Button`-дар). Компиляцияда
  мономорфтау; орындалу кезінде ешбір қосымша таратпа жоқ.
- **`Vec<Box<dyn Draw>>` (трейт-нысан)** — әр элемент әр түрлі типте
  болуы мүмкін. Компиляция кезінде мүмкіндік төмен; орындалу кезінде
  vtable арқылы динамикалық таратпа.

Қайсысын таңдау? Жобаға қарай:

- Тек бір типтегі ұжым жеткілікті болса — жалпылама, ол жылдамырақ.
- Бірнеше типтерді біріктіру керек болса — трейт-нысан.

### Нысан-қауіпсіз трейттер

Барлық трейт трейт-нысан бола алмайды. Тек **нысан-қауіпсіз**
(object-safe) трейттер. Қарапайым ережелер:

1. Трейттің барлық әдістері `&self` немесе `&mut self`-пен басталу
   керек.
2. Әдістерде жалпылама тип параметрлері болмауы керек.
3. Әдіс `Self`-ті қайтару керек емес.

Мысалы, `Clone` трейтінің `clone(&self) -> Self` әдісі — `Self`-ті
қайтарады. Сондықтан `Box<dyn Clone>` мүмкін емес.

Компилятор сізге айтады:

```text
error[E0038]: the trait `Clone` cannot be made into an object
```

Шешім — трейтті бөлектеу: бір нысан-қауіпсіз бөлік, бір емес.

## 17.3. Объектілі-бағытталған жобалау үлгісін жасау

Бір нақты ОО-үлгісі — **күй үлгісі** (state pattern). Ол: бір нысанның
мінез-құлқы оның ішкі күйіне байланысты ауысуын білдіреді. Күй —
жасырын; сыртқы код тек интерфейсті қолданады.

Мысалы, блог пост циклы:

1. **Жоба** (Draft) күйі — пост құрылды, жариялауға дайын емес.
2. **Тексеруге** (PendingReview) күйі — автор тексеруді сұрады.
3. **Жарияланған** (Published) күйі — пост жарияланды, барлығы көре
   алады.

Әр күйде нысанның мінез-құлқы әр түрлі: «жариялау» командасы тек
**Жарияланған** күйде ғана нақты тиімді. Әр күйдің өз ережесі.

### Дәстүрлі ОО-нұсқа: трейт-нысандармен

`Post` структы күй-нысанын ұстайды. Күй — `State` трейтін бекітетін
кез келген тип:

```rust
pub struct Post {
    state: Option<Box<dyn State>>,
    content: String,
}

impl Post {
    pub fn new() -> Post {
        Post {
            state: Some(Box::new(Draft {})),
            content: String::new(),
        }
    }

    pub fn add_text(&mut self, text: &str) {
        self.content.push_str(text);
    }

    pub fn content(&self) -> &str {
        self.state.as_ref().unwrap().content(self)
    }

    pub fn request_review(&mut self) {
        if let Some(s) = self.state.take() {
            self.state = Some(s.request_review());
        }
    }

    pub fn approve(&mut self) {
        if let Some(s) = self.state.take() {
            self.state = Some(s.approve());
        }
    }
}

trait State {
    fn request_review(self: Box<Self>) -> Box<dyn State>;
    fn approve(self: Box<Self>) -> Box<dyn State>;
    fn content<'a>(&self, post: &'a Post) -> &'a str {
        ""
    }
}

struct Draft {}

impl State for Draft {
    fn request_review(self: Box<Self>) -> Box<dyn State> {
        Box::new(PendingReview {})
    }

    fn approve(self: Box<Self>) -> Box<dyn State> {
        self
    }
}

struct PendingReview {}

impl State for PendingReview {
    fn request_review(self: Box<Self>) -> Box<dyn State> {
        self
    }

    fn approve(self: Box<Self>) -> Box<dyn State> {
        Box::new(Published {})
    }
}

struct Published {}

impl State for Published {
    fn request_review(self: Box<Self>) -> Box<dyn State> {
        self
    }

    fn approve(self: Box<Self>) -> Box<dyn State> {
        self
    }

    fn content<'a>(&self, post: &'a Post) -> &'a str {
        &post.content
    }
}
```

Тестілеу:

```rust
fn main() {
    let mut post = Post::new();
    post.add_text("Бүгін түскі асыма көк көжемен наны жедім.");

    assert_eq!("", post.content()); // Жоба — мазмұн көрінбейді

    post.request_review();
    assert_eq!("", post.content()); // Тексеруде — әлі көрінбейді

    post.approve();
    assert_eq!(
        "Бүгін түскі асыма көк көжемен наны жедім.",
        post.content()
    ); // Жарияланды — мазмұн көрінеді
}
```

Әр күй өз әдістерін бекіткен. Күй ауысқанда нысанның мінез-құлқы
автоматты түрде өзгереді.

`self: Box<Self>` ерекше: бұл әдіс шакырылғанда күй-нысанының
иелігін алады. Сондықтан күй ауысқанда — ескі күй-нысаны тазалынып,
жаңасы орнына шығады.

### Артықшылықтары мен кемшіліктері

ОО-нұсқаның артықшылықтары:

- Күй-нысандар бір-бірінен бөлек. Жаңа күйді қосу — жаңа структ +
  `State` бекіту.
- Күйді ауыстыру — бір орында, нысан ішінде; сыртқы код өзгермейді.

Кемшіліктері:

- Күйлер арасындағы байланыс қатты — `Draft::request_review` нақты
  `PendingReview`-ке өткізеді. Жаңа күй қосу барлық бар күйлерді
  өзгертуді талап етеді.
- Бірнеше әдістерде ұқсас шифрлеу — `if let Some(s) = self.state.take()`
  үлгісі қайталанады.

### `Rust`-стилінде күйді типтерде кодтау

`Rust`-тың типтік жүйесі күрделірек: біз күйді *типтермен* кодтай
аламыз. Әр күй — өзіндік типке.

```rust
pub struct Post {
    content: String,
}

pub struct DraftPost {
    content: String,
}

impl Post {
    pub fn new() -> DraftPost {
        DraftPost {
            content: String::new(),
        }
    }

    pub fn content(&self) -> &str {
        &self.content
    }
}

impl DraftPost {
    pub fn add_text(&mut self, text: &str) {
        self.content.push_str(text);
    }

    pub fn request_review(self) -> PendingReviewPost {
        PendingReviewPost {
            content: self.content,
        }
    }
}

pub struct PendingReviewPost {
    content: String,
}

impl PendingReviewPost {
    pub fn approve(self) -> Post {
        Post {
            content: self.content,
        }
    }
}
```

Енді әр күй — нақты тип:

- `DraftPost`-та — тек `add_text` және `request_review`. `content`
  әдісі жоқ.
- `PendingReviewPost`-та — тек `approve`. `add_text` де, `content`
  да жоқ.
- `Post` — соңғы күй; ол `content` ұсынады.

`request_review` бар нысанды (иелікімен) қабылдап, жаңасын қайтарады.
Сондықтан ескі `DraftPost` ары қарай қол жетімсіз — типтік жүйе оны
бекітеді.

Шакыру:

```rust
let mut post = Post::new();
post.add_text("...");
let post = post.request_review();
let post = post.approve();
println!("{}", post.content());
```

Бұл нұсқа қатарынан:

- **Жариялануға дайын болмаған пост** — `content` шакыру мүмкін емес;
  компилятор тыйым салады.
- Күйлер арасындағы ауысу — нақты типтегі функция шакыру.
- Ешбір `match` немесе `Option::take` керек емес.

### Қайсысы дұрыс?

ОО-нұсқа — динамикалық, кеңейтуге келе беретін. Жаңа күй қосуға —
жаңа структ + `State` бекіту жеткілікті, бар код өзгермейді.

`Rust`-стиль — статикалық, типтік жүйеде кодталған. Қателер
компиляцияда. Әр операция нақты күйде ғана. Бірақ жаңа күй қосу —
функциялардың қолтаңбаларын қайта жазу.

Жобаға не керек — соны таңдаңыз. `Rust` екеуіне де келеді.

## 17.4. Тарау қорытындысы

`Rust` — ОО-тіл емес, бірақ ОО-тілдердің көп ұғымдарын қолдайды:

- **Тұсаулау** — `pub` арқылы.
- **Көп пішінділік** — жалпылама типтер (статикалық) немесе
  трейт-нысандар (динамикалық).
- **Мұрагерлік жоқ**, бірақ оның практикалық пайдасы — әдепкі іске
  асырулар + трейт-нысандар арқылы.

ОО-жобалау үлгілерін `Rust`-та екі жолмен жасауға болады: дәстүрлі
ОО-стиль (трейт-нысандар) немесе `Rust`-тың типтік жүйесін қолдану
(күйді типтерге кодтау). Көп жағдайда екіншісі қателерді ертерек
ұстайды.

`Rust`-тың тұжырымдамасы: «парадигмаға тиісті болма; есепті ең
қауіпсіз әрі тиімді жолмен шеш». ОО — ең көп бағдарламалаушыға
таныс модель; бірақ `Rust` сізге одан да жақсы балама ұсынады
тағы өкінбейтін.

Келесі тарауда — **үлгілер мен сай келтіру** туралы кеңірек айтамыз.
Біз `match` мен `if let` ұғымдарымен таныс болсақ, енді үлгілердің
толық мүмкіндіктерін қарастырамыз: оларды функция параметрлерінде,
`while let`-те, `for`-да, `let`-те қолдану.
