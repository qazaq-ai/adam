# 7-тарау. Үлкейіп жатқан жобаларды бумалармен, сандықтармен және модульдермен басқару

Жоба өсіп бара жатса, кодты бөлектеу — оны кеңейтуге, тестілеуге және
қолдауға қажетті шарт. Бір файлда бар бағдарлама уақыт өте қиындай
түседі: оқу қиын, өзгерту қауіпті, жекелік шектеулі. `Rust`-та осы
мәселені шешуге арналған бірнеше қабат бар. Кішіден үлкенге қарай:

- **Бума** (package) — бір немесе бірнеше сандықты бір жобаға біріктіретін
  бірлік. Бумада `Cargo.toml` файлы болады.
- **Сандық** (crate) — компиляция бірлігі. Сандық не **екілік** (`bin`),
  не **кітапхана** (`lib`) бола алады.
- **Модуль** (module) — сандық ішіндегі кодты топтастыратын бөлік.
  Модульдер бір-бірінің ішіне ұя жасап, ағаш құрылымын құрайды.
- **Жол** (path) — модуль ағашындағы белгілі элементті атау тәсілі.

Барлығы Cargo басқаруына бағынады: ол сізге бума мен сандық құрылымын
ұсынады, тәуелділіктерді шешеді, құрастыру тәртібін орнатады. Тарауда
бір-бір бөлектеп қарастырамыз.

## 7.1. Бумалар мен сандықтар

**Сандық** дегеніміз — `Rust` компиляторына біруақытта берілетін код
жинағы. Сандық не екілік (бағдарлама) не кітапхана (қайта пайдалану үшін)
бола алады.

- **Екілік сандық** — `main` функциясы бар, орындалатын файлды құрастыруға
  арналған. Мысалы, бұған дейін біз жасаған `hello_world` мен
  `guessing_game` — екеуі де екілік сандықтар.
- **Кітапхана сандығы** — `main` функциясы жоқ, орындалатын файл
  құрастырмайды; оның мақсаты — басқа сандықтарға пайдалы код ұсыну.
  `rand` сандығы — нақты осы.

«Сандық» терминін біз стандартты кітапхана құжаттамасында кездестіреміз.
Кез келген сандықта **сандық түбірі** деп аталатын файл бар — компилятор
сол файлдан бастап барлық кодты табады.

**Бума** — бір немесе бірнеше сандықты біріктіретін бірлік. `Cargo`
бумаларды жасайды; бір бумада әрқашан `Cargo.toml` файлы болады. Бумада:

- **ең көп бір** кітапхана сандығы;
- **қалаған көлемде** екілік сандықтар;
- бірақ кем дегенде бір сандық — болуы керек.

`cargo new my_project` командасы жаңа екілік буманы жасайды:

```text
my_project/
├── Cargo.toml
└── src/
    └── main.rs
```

Cargo конвенциялары:

- `src/main.rs` — бумамен бірдей атауы бар екілік сандықтың түбірі.
- `src/lib.rs` — бумамен бірдей атауы бар кітапхана сандығының түбірі.
- `src/bin/` қалтасының ішіндегі әр `.rs` файл — қосымша екілік сандық.

Сондықтан бір бумада бір кітапхана + бірнеше екілік сандық бола алады:

```text
my_project/
├── Cargo.toml
├── src/
│   ├── lib.rs           // кітапхана сандығының түбірі
│   ├── main.rs          // негізгі екілік сандық
│   └── bin/
│       ├── helper.rs    // helper екілік сандық
│       └── analyzer.rs  // analyzer екілік сандық
```

Бұл құрылым Cargo-ның шартты келісімі (convention over configuration)
жетістігі: сіз файлды дұрыс орынға қоясыз — Cargo қалғанын біледі.

## 7.2. Жекелік пен аумақты модульдер арқылы басқару

**Модуль** — сандық ішіндегі кодты топтастыратын құрылым. Олар екі мақсатқа
қызмет етеді:

1. **Аумақты ұйымдастыру**: байланысты функцияларды, типтерді, тұрақтыларды
   бір атпен біріктіру.
2. **Жекелікті басқару**: қандай элемент сыртқа ашық, қайсысы тек ішкі
   қолданысқа арналған — соны нақты бекіту.

Жаңа кітапхана сандығын жасап, көрейік. `cargo new restaurant --lib`
командасы кітапхана сандығын жасайды. `src/lib.rs` файлында мынадай код
жазайық:

```rust
mod front_of_house {
    mod hosting {
        fn add_to_waitlist() {}

        fn seat_at_table() {}
    }

    mod serving {
        fn take_order() {}

        fn serve_order() {}

        fn take_payment() {}
    }
}
```

Бұл — мейрамхана типтес жобаның модульдік құрылымы. `front_of_house`
(«алдыңғы жұмыс») модулі ішінде `hosting` мен `serving` ішкі модульдері
бар. Әр модулде функциялар жарияланған.

Модульдер — иерархиялық. Сандықтың жасырын **түбір модулі** —
`crate` — болады. Жоғарыдағы код мынадай ағашты жасайды:

```text
crate
 └── front_of_house
     ├── hosting
     │   ├── add_to_waitlist
     │   └── seat_at_table
     └── serving
         ├── take_order
         ├── serve_order
         └── take_payment
```

Бұл ағаш — компилятордың модульдер мен олардағы функцияларды табу
картасы. `Rust`-та модуль ағашы әрдайым `crate`-тен басталады.

Модульдерді әр жерде жариялауға болады:

- **Сандық түбірінде** (`src/lib.rs`-та) `mod hosting;` сөйлемі
  компиляторды `src/hosting.rs` немесе `src/hosting/mod.rs` файлынан
  модулді іздеуге шақырады.
- **Басқа модуль ішінде** (`mod hosting { ... }`) — модулді сол жерде
  тікелей жариялайды (in-line).

## 7.3. Модуль ағашындағы элементтерге жол

Модуль ішіндегі функцияны, структты немесе тұрақтыны атау үшін біз
**жол** (path) қолданамыз. Жол екі түрлі болады:

- **Абсолюттік жол** — `crate` түбірінен басталады.
- **Салыстырмалы жол** — қазіргі модульден басталады және `self`, `super`
  немесе модуль атымен жалғасады.

Мысалы, `front_of_house::hosting::add_to_waitlist` функциясын
сандық түбірінде шақырайық:

```rust
mod front_of_house {
    mod hosting {
        fn add_to_waitlist() {}
    }
}

pub fn eat_at_restaurant() {
    // абсолюттік жол
    crate::front_of_house::hosting::add_to_waitlist();

    // салыстырмалы жол
    front_of_house::hosting::add_to_waitlist();
}
```

Біздің кодты компиляция жасасақ — қате аламыз:

```text
error[E0603]: module `hosting` is private
```

Бұл — `Rust`-тың жекелік ережесінің бірі: **барлық элементтер әдепкі
бойынша жекеше**. Сыртқы модуль ішкі модульдің жасырын элементтерін
көре алмайды. Ішкі модуль өзінің сыртқы модулін көре алады, бірақ керісінше
жоқ.

Қажетті элементті ашық қылу үшін `pub` кілтсөзін қосамыз:

```rust
mod front_of_house {
    pub mod hosting {
        pub fn add_to_waitlist() {}
    }
}

pub fn eat_at_restaurant() {
    crate::front_of_house::hosting::add_to_waitlist();
    front_of_house::hosting::add_to_waitlist();
}
```

Енді `hosting` модулі сыртқа ашық, `add_to_waitlist` функциясы да ашық.
Бірақ маңызды нәрсе: `pub` бір қабат қана ашады. Модулді `pub` ету —
ішіндегі бәрін автоматты түрде ашпайды. Ішіндегі функцияларды бөлек
`pub` ету керек.

### `super` арқылы салыстырмалы жол

Кейде ағашта бір қабат жоғары шығу керек. `super` — «бір қабат жоғарыға»
дегенді білдіреді. Үлкен мысалда:

```rust
fn deliver_order() {}

mod back_of_house {
    fn fix_incorrect_order() {
        cook_order();
        super::deliver_order();
    }

    fn cook_order() {}
}
```

`back_of_house` модулі сыртында тұрған `deliver_order` функциясын
шақыру үшін `super::deliver_order()` дейміз. `super` — `back_of_house`-тан
бір қабат жоғары — яғни сандық түбірі. Сондықтан `super::deliver_order`
және `crate::deliver_order` — бір нәрсе.

`super` пайдалы, өйткені модуль ішкі қайта орналастырылған кезде көп
жолды қайта жазудың керегі болмайды.

### Ашық структ пен ашық енам

`pub` сөйлемі структ пен енамға да жатады, бірақ оларда нюанс бар.

**Ашық структ** ұқсас жағдайда тұрады: оны `pub` ету — өрістерін
автоматты ашпайды. Әр өрісті жеке `pub` ету керек:

```rust
mod back_of_house {
    pub struct Breakfast {
        pub toast: String,
        seasonal_fruit: String,
    }

    impl Breakfast {
        pub fn summer(toast: &str) -> Breakfast {
            Breakfast {
                toast: String::from(toast),
                seasonal_fruit: String::from("шабдалы"),
            }
        }
    }
}

pub fn eat_at_restaurant() {
    let mut meal = back_of_house::Breakfast::summer("қарабидай");
    meal.toast = String::from("бидай"); // toast — ашық, өзгертуге болады
    // meal.seasonal_fruit = ...        // ҚАТЕ: жасырын
}
```

`Breakfast::summer` байланысты функциясы құрылыс-конструктор ретінде
қызмет етеді — себебі `seasonal_fruit` жекеше, оны сырттан тікелей
тағайындау мүмкін емес. Сондықтан кітапхана өзіне қажетсіз жекелікті
ұстай отырып, пайдалы интерфейсті ашады.

**Ашық енам** — структтан өзгеше: `pub enum Foo { A, B }` дейміз — оның
**барлық нұсқалары** автоматты түрде ашық. Бұл табиғи: енамның мақсаты —
нұсқаларын ашу, оны жасыру семантикалық мағынасыз.

```rust
mod back_of_house {
    pub enum Appetizer {
        Soup,
        Salad,
    }
}

pub fn eat_at_restaurant() {
    let order1 = back_of_house::Appetizer::Soup;  // дұрыс
    let order2 = back_of_house::Appetizer::Salad; // дұрыс
}
```

## 7.4. `use` арқылы жолдарды аумаққа енгізу

Жолдарды толық жазу — қайталаушы жұмыс. `eat_at_restaurant` функциясын
қарасақ, әр шақыруда `front_of_house::hosting::add_to_waitlist()` деп
жазып отырамыз. Қысқартуға арналған құрал — `use` сөйлемі. Ол модульдік
жолды ағымдағы аумаққа қысқа атпен енгізеді:

```rust
mod front_of_house {
    pub mod hosting {
        pub fn add_to_waitlist() {}
    }
}

use crate::front_of_house::hosting;

pub fn eat_at_restaurant() {
    hosting::add_to_waitlist();
    hosting::add_to_waitlist();
    hosting::add_to_waitlist();
}
```

`use crate::front_of_house::hosting;` арқылы біз `hosting`-ты ағымдағы
аумаққа енгіздік. Енді тек `hosting::add_to_waitlist()` деп жаза аламыз.

`use` арқылы импортталатын элементтер аумақтың ережелеріне бағынады:
басқа модульде сол `use` сөйлемі қолжетімсіз.

### Идиоматиялық `use`

`use` сөйлемінде функцияны тікелей емес, оның ата-модулін енгізу —
идиоматиялық:

```rust
// Идиомалық
use crate::front_of_house::hosting;
hosting::add_to_waitlist();
```

Бұл функцияны шақырғанда оның қай модульдікі екенін көріп тұруға мүмкіндік
береді. Олай етпей тікелей функцияны импорттайтын болсақ:

```rust
// Идиомалық емес
use crate::front_of_house::hosting::add_to_waitlist;
add_to_waitlist();
```

— бұл шарт қарап оқуға қиынырақ.

Бірақ структ, енам мен басқа аталған типтерді тікелей импорттау —
идиоматиялық. `HashMap`, `String`, `Vec` сияқты:

```rust
use std::collections::HashMap;

fn main() {
    let mut map = HashMap::new();
    map.insert(1, 2);
}
```

### `as` арқылы атау өзгерту

Егер екі модульден бірдей атаумен типтер импорттасаңыз — соқтығыс пайда
болады. Шешімі — `as` арқылы атауды ауыстыру:

```rust
use std::fmt::Result;
use std::io::Result as IoResult;
```

Енді бір аумақта екі түрлі `Result` бар: біріншісі — `fmt::Result`,
екіншісі — `IoResult`.

### `pub use` арқылы қайта экспорттау

`use` сөйлемі әдепкі бойынша **жекеше**: импорттаған атау тек ағымдағы
аумақта көрінеді. Бірақ `pub use` дегеннен кейін атау ашық болады — оны
сыртқы код көреді. Бұл — **қайта экспорттау**:

```rust
mod front_of_house {
    pub mod hosting {
        pub fn add_to_waitlist() {}
    }
}

pub use crate::front_of_house::hosting;
```

Енді сыртқы код `restaurant::hosting::add_to_waitlist()` деп шақыра
алады — `front_of_house` модулін көре алмаса да. Қайта экспорттау
кітапхананың сыртқы интерфейсін ішкі құрылымынан бөлуге мүмкіндік береді.

### Сыртқы бумаларды қолдану

Сыртқы сандықты қосу үшін `Cargo.toml` ішіне тәуелділікті жазып, `use`
арқылы аумаққа енгіземіз. `rand` сандығын мысалға алсақ:

```toml
[dependencies]
rand = "0.8.5"
```

```rust
use rand::Rng;

fn main() {
    let secret = rand::thread_rng().gen_range(1..=100);
}
```

Стандартты кітапхана `std` де сыртқы сандық — бірақ онымен жұмыс істеу
үшін `Cargo.toml`-ге қосу керек емес: ол әрқашан қолжетімді. Мысалы:

```rust
use std::collections::HashMap;
```

### Тоғыспалы жолдар

Бір модульден бірнеше элементті импорттасаңыз — тоғыспалы жол синтаксисі
ыңғайлы:

```rust
// Тізбекті
use std::cmp::Ordering;
use std::io;

// Тоғыспалы — тең мағынасы
use std::{cmp::Ordering, io};
```

`self` арқылы ата-модульдің өзін де қосуға болады:

```rust
use std::io;
use std::io::Write;

// Тоғыспалы
use std::io::{self, Write};
```

### `glob` оператор

Бар бір модульдің барлық ашық элементтерін бірден импорттау керек болса
— `*` (glob) операторы:

```rust
use std::collections::*;
```

Бұл сирек қолданылады, өйткені кодты оқыған адам қандай атаулар
импортталғанын анық көре алмайды. Тестерде немесе арнайы `prelude`
модулінде ыңғайлы.

## 7.5. Модульдерді әртүрлі файлдарға бөлу

Жоба үлкейген сайын барлық модульді бір файлда ұстау қолайсыз. Әр
модулді өз файлына бөлуге болады.

Бар мысалымызды бөлектейік. `src/lib.rs` ішіне мынадай:

```rust
mod front_of_house;

pub use crate::front_of_house::hosting;

pub fn eat_at_restaurant() {
    hosting::add_to_waitlist();
}
```

`mod front_of_house;` — бұл «модулдің денесі осы файлда емес, басқа жерде
табу керек» дегенді білдіреді. Компилятор екі жерден іздейді:

- `src/front_of_house.rs` — жалпы файл;
- `src/front_of_house/mod.rs` — ескі стиль (қалта + `mod.rs`).

`src/front_of_house.rs` файлын жасап, ішіне жазайық:

```rust
pub mod hosting;
```

Және `src/front_of_house/hosting.rs` файлы:

```rust
pub fn add_to_waitlist() {}
```

Қалта құрылымы:

```text
src/
├── lib.rs
├── front_of_house.rs
└── front_of_house/
    └── hosting.rs
```

Бұл жұмыс істейді. Файлдарды бөлектегеннен кейін бағдарламаның
семантикасы өзгермейді — модуль ағашы дәл сондай. Бөлу — таза
ұйымдастырушылық.

Үлкен жобаларда модульдердің тереңдігі үш-төрт қабатқа жетуі мүмкін.
Cargo, `pub`, `use` мен модуль ағашы бар болғанда — ондай жобалар
жинақы әрі айқын ұйымдастырылады.

## 7.6. Тарау қорытындысы

`Rust`-тың модульдік жүйесі төрт қабат: бума, сандық, модуль, жол.
Cargo бума мен сандықты басқарады; модульдер сандық ішіндегі кодты
ағашқа топтайды; жолдар арқылы біз элементтерге қол жеткіземіз; `pub`
арқылы жекелікті, `use` арқылы импортты басқарамыз.

`pub use` (қайта экспорттау), тоғыспалы жолдар, `as` арқылы атау
өзгерту, `super`, `self` — бұлар модульдік жүйенің ыңғайлы құралдары.
Біршама үйренгенше, көп нәрсе танымайтын болып көрінуі мүмкін; бірақ
бірнеше нақты жобаны ұйымдастырып шыққанда олардың табиғи болып
келетінін көресіз.

Келесі тарауда стандартты кітапхананың ұжымдық типтерімен — `Vec`,
`String`, `HashMap` — танысамыз. Олар әр `Rust` бағдарламашысының күн
сайынғы құралдары.
