# 9-тарау. Қатені өңдеу

`Rust` бағдарламашысының қателіктерге деген қарым-қатынасы — оның тілдің
қалған дизайн шешімдеріне ұқсас: айқын, толық, әрі компиляция кезінде
тексеретіндей. Көп тілдер ерекше жағдайды (exception) қолданады және
қателі функцияны қалай шақыруды бағдарламашының есіне сақтайды. `Rust`
басқаша: қате типтік жүйеде көрінеді, оны қарастырмауға — компилятор
сізге жол бермейді.

`Rust`-та қателер екі үлкен топқа бөлінеді: **қалпына келтірілмейтін**
және **қалпына келтіруге болатын**. Қалпына келтірілмейтін қателер —
бағдарламаның жалғаспауы керек жағдайлар (мысалы, жиым шегінен тыс
индекстеу). Олар үшін `panic!` макросы бар. Қалпына келтіруге болатын
қателер — күтілетін жағдайлар (мысалы, файл табылмады, желі үзілді).
Олар үшін `Result<T, E>` түрі бар.

Бұл тарауда екеуінің қашан қолданылатынын, қалай жазылатынын және `?`
операторы арқылы қатені шақырушыға қалай таратуға болатынын қарастырамыз.

## 9.1. `panic!` арқылы қалпына келтірілмейтін қателер

Кейде кодыңызда жөнделмейтін жағдай туады. `Rust` сізге `panic!` атты
макросты ұсынады. `panic!` шақырылғанда бағдарлама қате туралы хабарламаны
басып шығарады, стек жадын кері айналдырады (cleanup жасайды) және
тоқтайды.

Қарапайым мысал:

```rust
fn main() {
    panic!("Apatқа барамыз!");
}
```

Іске қосылғанда:

```text
thread 'main' panicked at 'Apatқа барамыз!', src/main.rs:2:5
note: run with `RUST_BACKTRACE=1` environment variable to display a backtrace
```

Қандайда бір кодтан `panic!` шақырылса — бағдарлама солай тоқтайды.
Көрнекілеу нақты қатеге келтіретін жоғары деңгейлі мысал — жиымның
шегінен тыс оқу:

```rust
fn main() {
    let v = vec![1, 2, 3];
    v[99]; // panic!
}
```

C тілінде бұл қате анықталмайтын тәртіп — басқа жадтың бөлігін оқып
кетуі мүмкін, әрі бағдарламаны бұзып, кеселден тыс мінез-құлық тудыруы
ғажап емес. `Rust` бұған тыйым салады: индекс шегінен шықса — `panic!`,
ешбір жадтың бөлігіне зиян келмейді.

### Шегініс ізін көру

`RUST_BACKTRACE=1` атты ортаға айнымалы орнатсаңыз — `panic!` болғанда
бүкіл қоңырау тізбегі (стек ізі) шығарылады:

```sh
$ RUST_BACKTRACE=1 cargo run
```

Шегініс ізі қайдан шыққанын анық көрсетеді — қандай функциядан, қандай
кітапханадан. Жөндеу барысында бұл — өте пайдалы.

### `panic!` пен `abort`

Әдепкі бойынша `panic!` болғанда `Rust` стекті **кері айналдырады** —
функция-функциядан сатып шығып, әр айнымалыны `drop` арқылы тазартады.
Бұл — қауіпсіз, бірақ бинарлық файлды үлкейтеді (өйткені `Rust` стекті
қалай айналдыруды білуі керек).

Кейбір жағдайларда (ұзартпайтын жоспарланған бағдарламалар, ендірілген
жүйелер) сізге кішірек бинарлы файл маңызды. Сонда `Cargo.toml` ішіне
жазасыз:

```toml
[profile.release]
panic = "abort"
```

`abort` — стекті айналдырмайды, бірден тоқтайды. Жадты операциялық жүйе
босатады. Бұл — арнайы қажетсіз, әдепкі (айналдыру) қалпы — көп жағдайға
дұрыс таңдау.

## 9.2. `Result<T, E>` арқылы қалпына келтіруге болатын қателер

Көп қателер апаттық деңгейге жетпейді. Файл табылмаса — басқасын
қарастыруға болады. Желі үзілді — қайта тырысамыз. `Rust` бұл жағдайларға
`Result<T, E>` атты енамды ұсынады:

```rust
enum Result<T, E> {
    Ok(T),
    Err(E),
}
```

`T` — сәтті жағдайдағы мән; `E` — қате типі. Файлды ашатын `File::open`
функциясы `Result<File, std::io::Error>` қайтарады:

```rust
use std::fs::File;

fn main() {
    let greeting_file_result = File::open("hello.txt");

    let greeting_file = match greeting_file_result {
        Ok(file) => file,
        Err(error) => panic!("Файлды ашу мүмкін болмады: {:?}", error),
    };
}
```

`match` арқылы біз нәтижені тексереміз: `Ok(file)` болса — файл алып,
жалғастырамыз; `Err(error)` болса — `panic!`-қа барамыз. Бұл — қарапайым
бірақ дөрекі. `panic!` алдында қате түрін тексеріп, басқаша өңдеу де
мүмкін.

### Қате түрін бөлектеп өңдеу

Қателер әр түрлі — біреуі файл жоқ болғандықтан болуы мүмкін, екіншісі
рұқсат жоқ болғандықтан. Әрқайсысын басқаша өңдейік:

```rust
use std::fs::File;
use std::io::ErrorKind;

fn main() {
    let greeting_file_result = File::open("hello.txt");

    let greeting_file = match greeting_file_result {
        Ok(file) => file,
        Err(error) => match error.kind() {
            ErrorKind::NotFound => match File::create("hello.txt") {
                Ok(fc) => fc,
                Err(e) => panic!("Файл жасау мүмкін болмады: {:?}", e),
            },
            other_error => {
                panic!("Файл ашу мәселесі: {:?}", other_error);
            }
        },
    };
}
```

`error.kind()` — `std::io::ErrorKind` атты енам. `NotFound` болса —
файлды құрастырамыз; басқа қате болса — `panic!`. Кодтың тұрқы тым
терең, оқуға қиын. `Result`-та қателерді өңдеуге арналған әдістер бар —
оларды қолдану әдетте ыңғайлырақ.

### `unwrap` пен `expect` — қысқа жолдар

Жасанды бағдарламалар, тестілер, прототиптер үшін кейде Result-ты толық
match-пен өңдеу шектен тыс. Соған арналған қысқа әдістер:

- `unwrap` — `Ok` болса ішіндегі мәнді қайтарады; `Err` болса `panic!`.
- `expect("хабарлама")` — `unwrap`-қа ұқсас, бірақ `panic!` хабарламасын
  өзіңіз белгілеп бересіз.

```rust
use std::fs::File;

let greeting_file = File::open("hello.txt").unwrap();
let greeting_file = File::open("hello.txt").expect("hello.txt файлы болуы керек");
```

Прод-кодта `expect`-ті — `unwrap`-қа қарағанда — қолдану артық: қате
болғанда сіз нақты не күткеніңізді хабарлайсыз.

### Қатені шақырушыға тарату

Кейде функция өз ішінде қатені өңдемей, оны шақырушыға қайтару керек —
шақырушы шешсін. Бұл **қатені тарату** деп аталады. Қарапайым мысал —
файлдан атау оқу:

```rust
use std::fs::File;
use std::io::{self, Read};

fn read_username_from_file() -> Result<String, io::Error> {
    let username_file_result = File::open("hello.txt");

    let mut username_file = match username_file_result {
        Ok(file) => file,
        Err(e) => return Err(e),
    };

    let mut username = String::new();

    match username_file.read_to_string(&mut username) {
        Ok(_) => Ok(username),
        Err(e) => Err(e),
    }
}
```

Функция `Result<String, io::Error>` қайтарады. Әр операцияны `match`
арқылы тексеріп, қате болса оны бірден қайтарамыз. Шақырушы кодта
қатеге не істеу керек екенін шешеді — `panic!` ме, қайтадан тырысу ма,
басқа файлмен жалғастыру ма.

Қатені тарату бұндай жиі қажет болғандықтан, `Rust`-та оған арналған
арнайы оператор бар.

### `?` операторы

`?` операторы `Result`-пен жұмыс істейді. Орналасқан жерінде ол:

- `Ok(value)` болса — мәнді шығарып, өрнектің мәніне айналады.
- `Err(error)` болса — функциядан бірден `Err(error)` қайтарады.

Жоғарыдағы мысалды `?` арқылы жинақтайық:

```rust
use std::fs::File;
use std::io::{self, Read};

fn read_username_from_file() -> Result<String, io::Error> {
    let mut username_file = File::open("hello.txt")?;
    let mut username = String::new();
    username_file.read_to_string(&mut username)?;
    Ok(username)
}
```

Әлі қысқа жазуға болады:

```rust
use std::fs::File;
use std::io::{self, Read};

fn read_username_from_file() -> Result<String, io::Error> {
    let mut username = String::new();
    File::open("hello.txt")?.read_to_string(&mut username)?;
    Ok(username)
}
```

Стандартты кітапханада осы үлгінің ең қысқа нұсқасы да бар:

```rust
use std::fs;
use std::io;

fn read_username_from_file() -> Result<String, io::Error> {
    fs::read_to_string("hello.txt")
}
```

`fs::read_to_string` — файлды ашып, оны жолға оқып, файлды жабатын
дайын функция. Қателі әрбір қадамды өзіңіз жазудың керегі жоқ.

### Қатені түрлендіру

`?` операторы тек қатені тек қана таратпайды — ол қатені басқа қате
түріне **түрлендіре** алады, егер шақырушы күтетін қате типі мен `?`
қайтаратын қате типі арасында `From` трейті бекітілсе:

```rust
fn parse_age() -> Result<u32, MyError> {
    let s = std::fs::read_to_string("age.txt")?;
    let n: u32 = s.trim().parse()?;
    Ok(n)
}
```

Бұл функцияда екі түрлі қате болуы мүмкін: `io::Error` (файл оқу қатесі)
және `ParseIntError` (сан талдау қатесі). Екеуі де `?` арқылы `MyError`
түріне айналады, егер сіз `MyError` үшін `From<io::Error>` мен
`From<ParseIntError>` бекітсеңіз. 10-тарау мен 17-тарауда трейттер мен
қателерді бір типке жинау туралы толығырақ айтамыз.

### `?` қайда қолданылады?

`?` операторы тек белгілі бір типтерді қайтаратын функцияларда қолданылуы
мүмкін:

- `Result<T, E>` қайтаратын функцияларда — `?` қатені таратады.
- `Option<T>` қайтаратын функцияларда — `?` `None`-ды бірден қайтарады.
- `main` функциясында — егер ол `Result<(), Box<dyn Error>>` қайтарса
  (төменде көрсетеміз).

Сондықтан мұндай код жұмыс істейді:

```rust
fn last_char_of_first_line(text: &str) -> Option<char> {
    text.lines().next()?.chars().last()
}
```

`lines().next()` — `Option<&str>`. `?` арқылы `None` бірден шығады; `Some`
болса ішіндегі жол алынады. Содан кейін `chars().last()` тағы да `Option`.

### `main` ішінде `?`

Әдепкі бойынша `main` функциясы `()` (unit) қайтарады. `?` `main`-да жұмыс
істемейді. Бірақ `main`-нің қайтару типін `Result<(), Box<dyn Error>>`-ке
ауыстыруға болады:

```rust
use std::error::Error;
use std::fs::File;

fn main() -> Result<(), Box<dyn Error>> {
    let _greeting_file = File::open("hello.txt")?;
    Ok(())
}
```

`Box<dyn Error>` — кез келген қате типін қабылдайтын ұжымдық тип.
17-тарауда тереңірек қарастырамыз. Әзірге естеріңізде болсын: `main`-да
`?` қолдану үшін, қайтару типін осылай белгілеу керек.

## 9.3. Қашан `panic!`, қашан `Result`?

`Rust`-та қашан `panic!` шақыру керек, қашан `Result` қайтару керек —
қалай шешу керек? Жалпы ереже: **бұл қалпына келу мүмкін бе?**

- Егер сіз кодтың логикасында болмауы керек жағдайды кездестірсеңіз —
  `panic!`. Мысалы, индексі әрқашан жарамды деп күткен жерде ол жарамсыз
  болса.
- Егер бұл орындалу кезіндегі күтілетін қате болса (файл жоқ, желі
  қолжетімсіз, пайдаланушы дұрыс емес сан енгізді) — `Result`. Шақырушыға
  таңдау бер.

### Прототиптерде, тестілерде

Прототип немесе тест жазғанда — `unwrap` мен `expect` ыңғайлы. Олар сізге
қателерді өңдеу логикасына шектен тыс мән бермей, негізгі идеяны жаза
беруге мүмкіндік береді. Кейін тексеріп, нақты қателерді өңдейтін кодты
қосасыз.

### Сізге шарт жайлы көп білгіңіз келгенде

Кейде сіз компиляциядан тыс шарттардың орындалатынына сенімдісіз —
бірақ компилятор оны дәлелдей алмайды. Мысалы:

```rust
use std::net::IpAddr;

let home: IpAddr = "127.0.0.1"
    .parse()
    .expect("Қатты жазылған IP — әрқашан жарамды");
```

Бұл жерде сіз `"127.0.0.1"`-дің әрқашан жарамды IP болатынын білесіз.
`expect` сіздің жорамалыңызды құжат ретінде көрсетеді — әрі бір күні
қателескен болсаңыз — `panic!` сізге жарамсыз жорамалды табуға көмектеседі.

### Қашан `panic!` дұрыс таңдау

`panic!` дұрыс таңдау, егер:

- Кодтың логикасы бұзылды (типтік жүйе кепілдеген жағдай орындалмады);
- Жұмысты жалғастырудың қауіпсіз тәсілі жоқ;
- Тестпен тексеріле алмайтын постусловие бұзылды.

Стандартты кітапхана жиі `panic!`-қа барады: `Vec` индекс шегінен шықса,
`unwrap` `None`-ке тап болса, `divide_by_zero` орындалса. Бұл — кодыңыздың
логикасында қате бар деген сигнал, орындалу кезіндегі күтілетін шарт
емес.

### Қашан `Result` дұрыс таңдау

`Result` дұрыс таңдау, егер:

- Қате — күтілетін орындалу-кездегі шарт;
- Шақырушы қатені өңдеуге қабілетті болуы керек;
- Кітапхана интерфейсі — соның қолданушысы шешімді өзіне қалдырыңыз.

Кітапхана жазуда — `Result` нәтижесі. `panic!` пайдаланушыға таңдау
бермейді; `Result` оларға қатені бағдарламаның контекстіне сай шешуге
мүмкіндік береді.

### Жеке қате типтерін жасау

Үлкен жобаларда стандартты қателерді өзіңіздің домендік қатеңізге айналдыру
ыңғайлы. Мысалы, ойын бағдарламасы үшін:

```rust
pub struct Guess {
    value: i32,
}

impl Guess {
    pub fn new(value: i32) -> Guess {
        if value < 1 || value > 100 {
            panic!("Болжам 1 мен 100 арасында болуы керек, {} келді.", value);
        }

        Guess { value }
    }

    pub fn value(&self) -> i32 {
        self.value
    }
}
```

`Guess` структының жасырын `value` өрісі бар. Сыртта `Guess::new`
арқылы ғана жасауға болады, ал ол шектен тыс болса — `panic!`. Сонымен
кейінгі кодыңызда `Guess` алсаңыз — оның мәні әрқашан 1 мен 100 арасында
болатынына сенімді бола аласыз. Бұл — типтік жүйедегі инвариант. Шектен
тыс мән — кодыңыздың логикасында қате, сондықтан `panic!` дұрыс. Ал
пайдаланушы енгізген санды Guess-қа айналдырғанда — `Result` қайтарудың
жолы дұрыс (өйткені пайдаланушы қате енгізе алады).

## 9.4. Тарау қорытындысы

`Rust`-та қателер екі үлкен топқа бөлінген: қалпына келтірілмейтін
(`panic!`) және қалпына келтіруге болатын (`Result`). Қандайын таңдау —
шарттың табиғатына, жалғаса алу-алмауына, шақырушыдан күтілетін
жауапқа байланысты.

`Result<T, E>` — `Rust`-тың қате интерфейсі. `match` арқылы оны өңдеуге
болады; `unwrap` мен `expect` — қысқа жолдар; `?` — қатені таратудың
бағдарламашыға аса жайлы тәсілі. `From` трейті арқылы әр түрлі қате
типтерін бір ортақ типке аударып, `?` арқылы оларды бір қалыпта өңдеуге
мүмкіндік туады.

`Rust`-тың қате моделі алдымен қатаң болып көрінуі мүмкін, бірақ оның
артықшылығы — кодты оқыған адам функцияның қателеспелігін бір қарап-ақ
біледі. Енді қателерді ұмытып кетпейсіз — компилятор сізге естетіп
тұрады.

Келесі тарауда `Rust`-тың тағы бір күшті ұғымына — жалпылама типтер мен
трейттер — кірісеміз. Олар арқылы әр түрлі типке жарамды коды бір рет
жазып, көп ретте қолдана аласыз.
